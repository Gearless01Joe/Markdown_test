# 代码评审清单

## 评审角色

- **作者**：准备上下文、截图、测试结果，确保 PR 结构清晰。
- **Reviewer**：聚焦正确性、安全、性能、可测试性、文档同步。
- **Maintainer**：合并前确认 CI 通过、标签与版本信息。

## 检查清单

1. **需求一致性**：PR 描述是否对齐验收标准？是否影响其他模块？
2. **代码质量**：
   - 有无未使用变量、重复逻辑？
   - 是否遵循 [Python 风格基线](../guide/best-practices.md)？
3. **安全与性能**：
   - 输入校验、防注入、防越权。
   - 关键路径是否添加缓存/限流？有基准数据或统计吗？
4. **测试与验证**：
   - 单元/集成测试覆盖核心路径。
   - 演示截图、日志、`mkdocs build` 输出。
5. **文档与发布**：
   - 文档路径与导航已更新？
   - `update.md` 是否同步版本信息？

## 评审工具

- GitHub Review / Gitee Review
- `pre-commit` 自动评论（lint、格式化）
- `pytest --maxfail=1` + `mkdocs build` 作为 CI 必需步骤

## 通过与阻塞

- 「通过」需满足：CI 绿、至少 2 名 Reviewer 同意（核心模块），普通模块 1 名即可。
- 「阻塞」场景：破坏兼容性、缺文档、缺测试、潜在安全问题。

> 对于跨项目变更，建议在 `projects/index.md` 记录影响范围，并通知相关 Owner。

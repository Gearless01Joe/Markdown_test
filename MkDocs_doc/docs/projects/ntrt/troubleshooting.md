# 故障排查

本页记录 NTRT 项目常见问题及解决思路，便于快速定位并修复。

---

## 1. 数据库连接失败

**症状**
- 启动后抛出 `OperationalError`、`Can't connect to MySQL`
- `session_dict` 为空或找不到指定别名

**排查**
1. 确认 `application/settings.py` 中的主机、端口、账号、密码正确
2. 如果使用 VPN/堡垒机，检查网络链路是否通畅
3. 在目标机器上执行 `mysql -h ...` 以验证连接
4. 若使用 `medicine_test` 等本地库，确保 MySQL 服务已启动

---

## 2. JSON 解析/类型异常

**症状**
- 日志出现 “跳过 XX 条记录”
- `_process_records` 中大量 `skipped`

**排查**
1. `JSON_TYPE(field) != 'OBJECT'/'ARRAY'`：数据本身不符合预期，需要先清洗
2. 记录中含有 `NULL` 或空字符串，需在 `_fetch_records_with_condition` 添加更多过滤
3. 检查是否存在非法字符（如未转义的引号），尝试使用 `JSON_VALID` 筛选

---

## 3. 事务回滚 / 写入失败

**症状**
- 控制台提示 `session.rollback()`
- MySQL 未更新数据

**排查**
1. 查看 MySQL 错误日志，确认是否有字段类型或约束冲突
2. 核对 `_write_updates` 中传入的字段名与模型是否一致
3. 将 `batch_size` 调小，避免单次事务过大
4. 临时启用 SQLAlchemy 的 `echo=True` 以观察具体 SQL

---

## 4. 大量 “处理成功 0，跳过 N”

**场景**
1. 过滤条件过于严格，导致 `records` 为空
2. 处理函数返回 `None`（表示跳过）

**建议**
- 在 `_clean_dataset` 中打印 `extra_filters` 或 `records[:5]`，确认命中情况
- 逐步调试 `_process_breadth_search`、`_process_cited_articles`，确保返回结构正确
- 若需要保留原数据，可在处理器中 fallback 为 `return data`

---

## 5. 清洗耗时过长

**优化方向**
- 调整 `BATCH_SIZE`（例如 100 → 500）
- 将需要清洗的字段拆成多个任务并行运行
- 针对只需一次的字段，清洗完成后在数据库记录 “已处理” 状态，避免重复扫描

---

## 6. 日志过多或过少

**处理**
- 将 `print` 替换为 `logging`，并配置输出级别/文件
- 对于高频输出，可通过命令行参数或配置开关控制

---

## 7. ORM 模型找不到字段

**症状**
- 抛出 `AttributeError: type object 'NsfcTopicRcmdTaskList' has no attribute 'xxx'`

**排查**
1. 检查数据库结构是否与 ORM 定义一致
2. 如表结构发生变更，及时更新 `NsfcTopicRcmdModels.py`
3. 运行 `alembic` 或手工 SQL 以比对字段/类型

---

## 8. 定时任务重复执行

**现象**
- 同一批数据被清洗多次

**策略**
1. 在表中增加 “last_clean_at” 或 “is_cleaned” 字段，用以过滤
2. 将 `cleaning_plan` 拆分为“增量任务”“全量任务”
3. 利用外部调度器（Airflow/Celery）保证任务互斥

---

## 9. 如何定位单条异常记录

1. 在 `_process_records` 中捕获异常并打印 `record_id`
2. 使用 SQL `SELECT JSON_PRETTY(...)` 查看原始内容
3. 如果问题可重现，可将 `updates` 临时写入文件后人工校验
